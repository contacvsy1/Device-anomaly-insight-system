Roadmap

How this prototype can grow into a fuller anomaly intelligence system

The current code proves the core idea. The next improvements can add evaluation, richer baselines, better explanations, and user-facing review tools.

Recommended Upgrade Path

1

Add evaluation metrics

Use is_injected_anomaly to calculate recall, precision, false positives, and false negatives for the simulated spike rows.

2

Create group baselines

Compare each row against normal behavior for its DMA, device type, content type, and hour bucket.

3

Persist model and metadata

Save fitted preprocessing and model artifacts so detection behavior is reproducible across runs.

4

Add an LLM-backed insight layer

Replace or augment rules with a controlled LLM prompt that uses row values, baselines, and historical context.

5

Build a dashboard

Show trends, anomaly drilldowns, filters, model scores, and recommended action status in a web UI.

Production Considerations

Area Consideration
Data ingestionMove from simulation to scheduled batch input or streaming telemetry.
Data qualityValidate ranges, missing values, duplicate device records, schema changes, and timestamp consistency.
Model governanceTrack training data, feature definitions, model version, thresholds, and evaluation reports.
ObservabilityLog pipeline runs, anomaly volumes, execution time, failed writes, and data drift.
Action workflowRoute critical anomalies into tickets, alerts, Slack messages, or operational dashboards.
